异常处理

  项目现在还没有开始，但是有些内容我觉得先可以做一下简单的开发，异常处理就是其中一项。今天特意将之前所看的一些异常处理又翻看了一下，发现其实在SpringBoot中有很多机制能够自动完成异常处理，我之前使用的非常麻烦的Controller方式存在很多重复代码，虽然尽可能去减少这种情况的发生，但是还是让Exception中显得十分复杂。

  异常处理思路

  在WEB中，很多异常发生情况不可预估，可能由于用户输入一些不合法的数据而程序开发者并没有预估到导致，所以我们需要拦截这些异常并做日志记录，同时还要告知用户一些信息。

  最简单也是最常见的异常则是访问资源不存在，也就是我们常说的404异常。这个异常很多时候由于用户输入一些不正确的路径，或者我们的资源可能处于某种原因丢失所导致，我们需要返回给用户一个界面告知用户资源丢失。

  SpringBoot中提供了一些注解能够帮助我们非常方便快捷的完成这些异常处理，同时在其默认配置下能够访问我们所设定的一些资源。SpringBoot在检测到资源不存在时，会自动访问templates下的error包中的404页面，也就是说我们只需要在templates下建立error包同时建立404页面即可，不用再去费力的做一个类去访问404页面，同时500、error等页面也在该路径下，SpringBoot都会进行自动检索。

  @ ExceptionHandler

  这个注解是spring中异常拦截注解，当任何异常发生的时候，都会被该注解所标识的类所捕获，我们可以通过这个注解进行内部处理。

  这个类就是我们所设定的异常拦截类，当我们发现出现任何异常的时候都会进行拦截，拦截完成后，记录日志或者信息以便我们去处理修改。但是这里还有一个问题，如果我们希望当发生某些异常的时候能够跳转到某个页面，应该怎么做。虽然拦截器能帮我们拦截大部分异常，但是只能够跳转到某一个指定页面，所以这里我打算这样处理。

  定义异常

  首先来定义一个自定义异常，针对404，我们的异常就叫做NotFoundException，它继承自RuntimeException，一般来说RuntimeException足够我们使用了。

  @ ResponseStatus

  这个注解能够标识在方法和类之上。

  • 标识方法: 该方法一定会抛出异常，但是如果方法如果没有抛出异常时会正常运行下去。
  • 标识类: 一般标识自定义异常类，能够自动进行一些处理。

  该注解和ExceptionHandler配合使用效果极佳，能够在拦截所有异常的前提下放行一些异常的抛出，让特定的Controller去处理这些异常。下面我们来看一下在Handler中如何实现。

  在这里我们用一个Annotation工具来找到状态码，当发现其不为空的时候将其抛出，此时我们的另一个Controller将捕获改异常并处理跳转对应页面。

  异常信息记录如下

2019-04-05 19:10:07.090 ERROR 10004 — [nio-8080-exec-1] c.m.Controller.ExceptionController : RequestURL : http://localhost:8080/,异常信息 : 资源不存在!
2019-04-05 19:10:07.092 WARN 10004 — [nio-8080-exec-1] .w.s.m.a.ResponseStatusExceptionResolver : Resolved [com.magnolia.Exception.NotFoundException: 资源不存在!]

  我们并没有写跳转到404页面的语句，但是能够自动跳转404是由于我们标识了状态码，Spring会根据状态码跳转到一些特殊页面，同时我们也能在页面中定义一些信息，这些信息可以在ResponseStatus中通过reason字段来填写。

  总结一下，我们在处理异常时，有些异常只需要跳转到同一个页面，有些异常需要特定页面，但是我们需要将所有异常捕获，那么我们就可以使用ExceptionHandler注解来拦截所有异常，但是针对特殊异常需要进行特定处理时，通过状态码标签将其标注，并在异常拦截器中将其抛出，让指定的拦截器捕获处理即可，这样业务逻辑会比以前简单很多，而且实现起来也很快捷。

  数据库新需求

  这几天我们一直在做一些数据库的工作，我们希望做到一个数据库可以进行拓展，让其更加方便来使用，同时针对不同模型能够给予支持。由于这个项目最终可能会对接到非常多的小项目，所以我们必须有一张能够适配所有内容的标准表。

  这张表应该具有以下几个特点:

  • 能够兼容多个项目信息记录
  • 能够针对单一项目进行CRUD等操作
  • 便于入库以及出库的操作

  结合这几种情况，我们想把表改为字段模式，简单来说每个表中有八个字段，这八个字段只储存特定的、按照要求输入的一组值，但从数据库上来看我们不知道这组值能够干什么，但是之后我们会根据数据查询进行拓展，例如当我们查到这条数据是来自空气检测机构的数据，我们会在这个字段后进行单位匹配。

  关于如何匹配这一点我们打算做成配置文件模式，通过配置文件读取对应的单位来减少源码的更改。昨天晚上把数据库设计好了，但是今天早晨有来了个新的需求!(无语…..)

Mybatis逆向工程

  这次项目在逻辑层操作并不是太过于复杂，更多的主要在SQL语句的书写，但是!我SQL语句写的特别菜，所以打算用Mybatis的逆向工程。

  逆向工程

  Mybatis中逆向工程指能够通过表为我们自动生成Mapper、Model以及SQL语句，也就是说只要我们建好了一张表，然后通过配置把它导入Mybatis逆向工程中，我们就能获得这张表的增删改查数据。(实际上有一个骚操作，就是用JPA自动创建数据库，然后再用Mybatis逆向工程获得数据表，我们基本上不用对数据库进行任何操作就能获得CRUD内容)

  配置还是很简单的，XML配置文件以及一个启动类就能够完成配置。

  启动类

  一份非常详细的配置文件，以后可以参照这个来配置，非常方便。

  逆向工程中的一些问题

  在逆向工程中我们碰到了下面几个问题。

  • 会生成一个WithBLOBS的实体类，而且会有很多报错情况
  • 接口对应的XML异常复杂，很难使用

  在网上找了很多教程，大部分都是来回抄来抄去，但是大部分都提到了数据对应问题。首先我想到的是Mybatis既然要逆向生成必须经历以下几个步骤。

  • 获取数据库及对应表
  • 将MySql中的数据类型与Java中数据类型进行对应
  • 创建Mapper接口
  • 创建Mapper.XML

  这里问题就出在了第二步，框架并不清楚我们会将数据对应成什么样子的形式，一般来讲VARCHAR对应String，但是一些特殊情况可能VARCHAR对其他的一些情况。所以框架会给我们提供一个WITHBLOBS的类来完成数据对应

  如果我们不想，只需要将表中字段与JAVA中字段相对应即可完成，最好是全部对应，否则可能会出现一些问题。

Swagger

  Swagger是个很神奇的东西，虽然使用起来很简单，但不妨外它是个非常便捷的接口工具。实际上这是我第一次接触前后端分离开发，之前一直有听说，现在使用起来，感觉很棒。

  虽然这次更新主要说的是Swagger，但是实际上我更愿意去聊一聊这种新的(仅仅对我而言)开发模式的一些感受。

  先来简单说说什么是Swagger

  Swagger是一款前后端分离开发的接口文档，这样听上去有点晕。我们知道，前后端分离是现在的大趋势，随着用户的计算机运算能力越来越强，很多逻辑层处理都放在了前端执行，而我们后端则减少了很多压力，这样就要求我们前后端分离时能够提供明确的接口来处理。那我我们该如何做呢?

  以往我们会写文档，但是这样有很多不方便的地方，忘记修改，多人修改同一个方法会造成文档混乱，当时间紧迫时可能文档更新和接口更新速度不一致等等。而Swagger解决了这些痛点。

  Swagger通过注解的方式将文档中出现的一些内容与方法绑定，换句话说，在你修改某个方法是，Swagger就帮你完成了一些文档的更新，这点尤其符合现在的RESTful标准。

  使用起来有以下几个步骤

  • Maven引入Swagger的包以及UI，由于Swagger自己内置了一个UI，用起来很方便，所以一般都会将该UI引入

  • 配置Swagger一些初始化的内容

  • 在Controller层添加一些Swagger的注解来表示信息

  此时运行服务器，然后打开localhost:8080/swaggeriui.html 就能够看到接口文档

  可以看到在方法上的注解实际上就是文档中对应的一些内容，同时更多注解对应请参考官方文档。

  这种体验对于后端程序员来说很友好，对比之前的JSP，Freemark等页面渲染技术来说，这种分离的感觉能让你更加专注与后端逻辑，而非不断的考虑前后端是否能够获取到值，能否在前端成功展示等等。
