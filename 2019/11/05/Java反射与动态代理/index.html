Java反射与动态代理

发表于 2019-11-05 更新于 2020-04-26 分类于 Web框架

  Java反射机制是SpringIoC技术实现的基础，同时也是拓展很多功能的重要方式。简单来讲，反射技术允许我们在程序运行的过程中，随意获得任意一个类的信息(属性与方法)，调用任意一个对象的方法。

  什么是反射？

  Java是一门面向对象的语言，对象作为Java世界中最小的执行对象，能够承担各种事务。但是其中有两部分不属于面向对象的范畴。

  • 基本数据类型 基本数据类型不属于面向对象(后Java为其提供包装类)
  • 静态 被静态修饰的所有方法、变量都是通过类名直接调用，并非面向对象。

  而Java中为我们提供了一个名为Class的类，该类非常特殊。如果说对象都属于特定的类，例如person类的对象属于person类，那么我们自己创建的所有类，都属于Class类。也就是说，person类是Class类的一个对象。这里对象并非我们所简单理解的new关键字所创建的对象。而是在概念上的一个对象，也就是说，所有通过class关键字所创建的类，都属于Class的对象。通过最简单的例子我们可以看一下。

  可以看到，直接使用类名.class方法能够创建Class的对象，通过这个例子就很容易理解我们所创建的类都是Class类的子类，而Class类的引用官方称作为类类型(class type)。同时，Class类无法创建实例对象，这点在源码中就能看到。

  在了解反射之前，我们需要先对于Java语言编译运行原理进行简单的了解。Java语言也是一门半解释性语言，我们在编译器编写的代码后缀都是.java格式，而JVM所运行的则是.class后缀文件，JVM将.class字节码解释为机器码在计算机上进行运行。

  在Java中，类加载是十分复杂的，但是有一个最为简单的原理，JVM在程序运行时并不会加载所有的类，而是当你使用到该类的时候，才会加载该类，此时如果该类中又依赖着其他的类，则优先加载依赖类，直到加载到一个独立类(该类并不依赖任何类，可以直接加载，例如我们的实体类)。这种加载方式称作为静态加载方式，但是这种方式会让程序过于固化，一切过程都在程序Run那一刻保持不变，如果我们能够在程序运行的过程中，动态的加载某个我们想要的类，来实现一些方法，这样能让程序的灵活性有了巨大提升(看起来十分类似AOP的实现)，而Java中则通过反射的方式完成，这种类加载方式则是动态加载。

  如何使用反射?

  使用Class类来获得对象的三种方法。

  Java中反射为我们提供了三种通过Class获得对象的方法，而在实际中，前两种方法使用相对较少，第一种已经获得了对象引用再去通过getClass方法获取使用一样不大，第二种获取对象方式与new关键字获取差别并不大。而第三种通过类全路径名称获取则是重点，我们可以通过输入的方式去动态加载某个类，我们简单来实现一段代码。

  可以看到，我们通过这种方式能够模拟工厂模式，动态加载我们所需要的类，再使用多态即可完成对于类型的转换。

  而反射中，我们能够动态使用类的方法，我们先从构造器来讲。反射机制中，使用方法都需要先获得方法的对象，之前为Method，构造器则为Constructor，通过构造器对象调用一些方法我们遍能够完成构造函数的使用。

  由于一个类中拥有多个构造器，同时可能有private、public等修饰，所以Constructor为我们提供了多个方法来快速获取不同的构造器。

  • getConstructors(): 获取所有公共(public)构造方法
  • getDeclaredConstructors: 获得所有构造方法
  • getConstructor(null): 获得共有无参数构造方法
  • getConstructor(参数1.class,参数2.class): 获得如要求属性参数的构造方法

  成员变量的获取以及使用

  成员变量的获取和使用和构造方法类似。

  • getFields: 获取所有公有的字段
  • getDeclaredFields: 获取所有的字段
  • getField(“属性”): 获取指定公有目标字段
  • getDeclaredField(“属性”): 获取指定私有字段

代理模式(Proxy)

什么是代理模式?

很多人可能总结的都不太一样,我个人理解中,代理模式是一种委托机制。举个例子: 当我们需要让一个人完成一些任务的时候,这个人能力不够,那么我就需要找一个代理人来帮助我们完成。

在开发中,我们可能每个阶段对一个类的需求都不一样,拿日志功能举例 :

阶段一 : 我们需要记录用户的登录信息等内容
阶段二 : 我们需要记录了用户操作了哪些内容
阶段三 : 需求更改,我们需要记录用户好友相关内容

那么开发过程中就会频繁对于日志类进行修改,不断的增添代码。但是假如哪天需求突然变更,这些新的功能不需要了,那么我们又要更改这些代码,这样就会很麻烦。

那么有什么好的解决办法?

静态代理

刚才的场景可以用静态代理来解决,通过静态代理,我们能够创建一个代理类,对我们的日志类功能完成增强,如果不需要增强,将代理类屏蔽即可。

静态代理是最容易理解实现的代理模式,首先我们先来规定三个个定义。

  • 目标类 : 我们需要进行增强的类
  • 代理类 : 增强后的类
  • Proxy接口 : 二者共同的接口

这里使用Proxy接口主要为了使用Java提供的多态,这样能够完成功能增强。

静态代理优劣

优点 : 编写简单,容易理解
缺点 : 代理类与目标类必须实现相同的接口，随着需要代理的目标类增多,接口会增多,同时代理类也会增多,维护性会变差。

为了解决以上这个问题,学者们提出了动态代理的思想。

动态代理

何为动态代理?

动态代理是更具复用性的一种代理方案,在静态代理中,我们需要在代理类内部维护着目标类的接口引用,随着目标类的增多,可能会有几十个接口引用,这样必然带来代码可多性差,冗余的问题。

动态代理为我们提供了一种新的思维模式,是否代理类不需要实现接口就能够完成对于目标类的增强?这种方案也意味着更加低的耦合性，能够让程序进一步降低依赖。

在Java中一般用到两种代理模式 :

  1. JKD动态代理 : 需要目标类实现一个接口
  2. Cglib动态代理 : 需要目标了可以继承(也就是说目标类不可用final字段标识)

Cglib动态代理

虽然JDK动态代理已经很方便了,但是还是需要为目标类实现一个接口,如果我们不想实现接口,那么可以使用Cglib动态代理。

Cglib动态代理使用的是子类继承的方式进行增强。我们知道在子类中可以使用super关键字调用父类方法,Cglib就是使用了这个特点。那么Cglib对于目标类也有一定的限制。

  • 目标类不可以是final类,final无法被继承
  • 目标类的final方法不可以被重写,无法增强
