Spring起源概括及IOC基本使用

  Spring框架是目前JavaEE中最为优秀的一款框架，其开源、轻量、模块化等优势能够轻松搭建一站式企业系统。相对于曾经出现的许多侵入式框架，Spring拥有模块化，非侵入式特点，我们能够选择组件去构成项目，十分方便。

  Spring是一个开放源代码的设计层面框架，它解决的是业务逻辑层和其他各层的松耦合问题，因此它将面向接口的编程思想贯穿整个系统应用。Spring是于2003 年兴起的一个轻量级的Java 开发框架，由Rod Johnson创建。简单来说，Spring是一个分层的JavaSE/EE full-stack(一站式) 轻量级开源框架。

  一句话来说:Spring是一个轻量级的控制反转(IoC)和面向切面(AOP)的容器（框架）。

  下面给出Spring所需要的组件及依赖下载地址。

  • 官网： Spring官方网站
  • 框架下载地址： SpringFramework下载地址(各个版本)
  • Spring框架文档：Spring框架组件官方文档
  • Spring框架源码：SpringFramework源码(Github)

Spring框架功能

  轻量: 与其他框架相比较，Spring在开销与框架本身大小上足以称得上轻量级，原生SpringFramework大小不超过1MB，同时框架设计为非侵入式，Spring中的对象并不依赖于Spring框架中的某特定类。

  IOC(控制反转): Spring框架实现IOC控制反转，降低程序间的耦合。Spring通过XML或者注解的方式实现IOC，取消了以往对象依赖时需要自己主动创建，而是通过在对象初始化时就把依赖进行注入。

  AOP(面向切面编程): Spring内部实现面向切面编程。分离业务逻辑和系统逻辑(例如注册服务和日志服务)，通过面向切面编程完成对于OOP的补充，进一步降低程序的耦合，提高程序内聚。

Spring框架组成

  Spring框架中主要定义了7个模块，其中其他模块构建在核心模块之上。

  Spring Core: Spring中的核心模块，其中包含了Beans的创建，ApplicationContext工厂模式的实现，主要将对象的创建及依赖注入与程序代码分离。

  AOP(切面模块): AOP与Aspects模块共同组成Spring中面向切面编程的模块，其中封装了AOP的使用规范，日志记录等内容依赖。

  Data模块: Spring Data Access模块中主要封装了对于持久层的一些服务，例如Spring中封装的JDBC工具，ORM服务框架等。其中ORM中包括Hibernate、iBatis等框架。

  Spring Web模块: Web模块中继承了Servlet、Socket，同时实现了对于Struct框架的集成。降低了Web请求以及多域之间对象存储管理的复杂度。

IOC控制反转

  控制反转Ioc(Inversion Of Control)，是一种设计思想，DI(依赖注入)是实现IoC的一种方法，也有人认为DI只是IoC的另一种说法。没有IoC的程序中我们使用面向对象编程对象的创建与对象间的依赖关系完全硬编码在程序中，对象的创建由程序自己控制，控制反转后将对象的创建转移给第三方，个人认为所谓控制反转就是：获得依赖对象的方式反转了。

  理想的系统为各个模块独立工作，其中不依赖于其他模块。而在实际中，我们一般无法做到如此的低耦合，常常会通过第三方的模块实现耦合降低，取代不同模块之间的耦合。

  IoC是Spring框架的核心内容，使用多种方式完美的实现了IoC，可以使用XML配置，也可以使用注解。一般来说，IOC运行主要经过以下步骤。

  • IOC核心容器加载
  • 读取XML配置文件
  • 根据元数据对Beans按照信息进行组装
  • 从IOC容器中获取Beans

什么是AOP？

  Aspect Oriented Programming(面向切面编程)：将有共同处理的业务，从传统的业务中抽离出来单独的封装，通过装配(注解)的方式作用于系统。在不修改系统业务逻辑的前提下给系统追加功能。

  以往我们拓展业务、方法需要通过继承、多态、接口等途径进行实现，但是这种方法同样有着一种缺点，通过以上方法，只能通过纵向实现拓展，导出类实现不同的接口，实现诸多方法。但是随着功能增加，其中代码量毕竟越来越多，越来越难维护。例如实现一个功能，需要有权限校验、服务以及日志记录等功能，这样只能让系统服务和业务逻辑混杂在一起，十分难以维护。以图片为例。

  使用AOP后，可以完成从横切面对方法的插入，我们只需要告知Spring框架我们需要在哪里去插入哪些方法，在连接点运行时，框架会使用动态代理为我们运行插入的方法。通过这样能够有效将少代码混杂，可以将不同业务的代码抽离，进一步降低耦合度。

  AOP是对于OOP(面向对象编程)的一个补充，面向对象编程允许我们使用继承，多态来完成纵向的拓展，AOP则补充了横向切面的程序拓展，二者的结合让程序的可拓展性大大提高。

AOP的重要概念

  在Spring框架为我们提供的AOP中，有以下几个重要的概念。

  • Aspect: 切面，一般为一个类，类中包含着切入点与通知
  • JoinPoint: 连接点，程序中运行明确的点，一般为某个方法
  • Advice: 通知，AOP在特定的切入点上的增强处理，有前置，后置，异常，最终，环绕五种
  • Pointcut: 切入点，AOP增强位置，需要书写切入点表达式

  AOP使用方法

  AOP使用之前先要检查框架jar包是否导入齐全。Spring中为开发者提供了两种使用方式，装配与注解，二者可以同时结合使用。

    • < aop:around >: 环绕方法，在切入点运行前、后都将进行执行，同时环绕方法也是aop中使用最广泛、最为重要的方法。

自动装配

  在Spring中，自动装配功能是一个能够减少我们XML文件编码量的功能，简单来说，自动装配是按照一定格式、名称对应，能够完成对于类中所依赖的属性自动进行注入。

  • < By-Type >:如果类中属性类型和bean中class相同则进行注入
  • < By-Name >:如果类中的属性名称和bean中id相同则进行注入
  • < By-Constructor >:通过构造器进行自动装配

  首先准备两个类，Service和Dao，在Service中维护着Dao的引用，通过该应用调用Dao层方法，模拟业务处理。

  按照以往的配置，可以使用ref标签将dao层的对象进行注入，而使用自动装配后，Spring会对需要自动装配的类进行检查，首先检查该类中属性名称，其检查方法为查找该属性的Set方法，然后将Set后面的名称首字母小写。检查该名称与XML配置中的Bean文件id是否有对应，如果有则自动将bean中的配置进行注入。

  如果为查找到则在调用方法时无法使用set进行注入，必然会出现空指针异常。而使用ByType自动装配时，框架则会寻找配置文件中与自动装配类中的成员变量同类的bean，然后将其注入。但是如果在XML文件中出现了多个同类的bean，框架则会抛出不唯一异常。

  其中通过构造器实现自动装配需要我们在实体类中提供构造方法，而在底层实际上也是通过ByType对构造参数进行装配，按照构造参数中提供的类型去扫描XML文件，对于符合要求的Bean自动装配.

  在头文件中可以声明全局的autowrite，声明之后对于整个XML文件中的所有bean都能够执行自动装配。

  自动装配固然能给XML进行一定程度的瘦身，但是相对来说，我更加偏向于使用一般的依赖注入，首先使用自动装配后有些类的实例创建需要被限制，不能出现多个实例，否则会出现不唯一的情况，其次会导致XML配置文件逻辑混乱，无法在第一时间了解不同类之间的依赖关系。

  自动装配也可以使用注解来进行实现，Spring2.5版本后引入注解，大部分的功能都能实现注解实现，但是需要引入SpringAOP的Jar包，由于我们使用IDEA引入的Spring框架，其中都包括了，这里就不再进行过多实现。

  • @ Autowired 其中required值中true表示该引用必须存在对象，false表示引用可以使用null。
  • @ Qualifier 必须在自动装配注解后使用，使用该注解后，将按照ByName的方式进行自动装配。
  • @ Resource 和@ Autowired的注解方式相似，只不过Resource先会按照ByName的方式进行，如果找不到符合的Bean，则会默认使用ByType的方式自动装配。

Bean的生命周期

  Spring中Bean的加载无疑是一项非常重点的内容，实际上Bean的加载十分复杂。我们来按照下面这张图来解释。

  首先根据对应，分别为实例化、设置属性值、调用setBeanName方法、调用setBeanFactory方法、设置ApplicationContext容器、调用预初始化方法，调用定制的初始化方法、调用后初始化方法，此时bean就能够被使用了。之后则是Bean的销毁的一些内容，调用预销毁方法，调用用户定制的销毁方法，然后Bean的生命周期结束。

  以上即为Bean在Spring中完整的一个生命周期，实际上，我们可以简单理解为一下四个步骤。

  • Bean的加载 通过XML进行Bean的配置
  • Bean的初始化 调用系统、或者我们自定义的初始化方法
  • Bean的使用 通过容器来获得Bean的实例，进行业务处理
  • Bean的销毁 销毁容器、或者Bean声明周期结束后调用销毁方法

  初始化和销毁方法在Spring框架中都为我们进行了基本的实现，我们可以通过多种方法来进行对于Bean的初始化和销毁处理，主要分为两种方式，一，通过实现Spring为我们提供的接口

  • InitializingBean接口 该接口中有afterPropertiesSet方法，实现该接口后重写该方法，在Spring初始化的时候会自动加载该方法中的代码，该方法即为预初始化方法。
  • DisposableBean接口 实现该接口后重写destroy方法，与初始化相同，该方法为预销毁方法。

  二是我们能够通过XML文件为某个类配置初始化或者销毁的方法。

  • init-method 初始化方法
  • destory-method 销毁方法

  这两个方法可以在Bean标签中进行配置，通过bean中配置，在Bean实例化时，会先进行初始化方法的加载，在销毁时调用destory方法。

  前两天学习Spring中的事务管理，发现能够连带复习到非常多的内容，甚至连SQL语句拼写都能够学习不少，今天想总结一下关于Spring的事务内容。

  Spring中提供了事务管理机制，依赖JDBC模板或者是继承的框架来完成，首先来看一下事务。

  最为经典的案例即为转账案例。首先拥有账户A、B，其中A需要向B进行转账，虽然转账是一个行为，但是实际上，在持久层至少需要两个操作来进行实现，首先对A账户的账目减少，然后再对B的账目增加相同的数值。对于Service层中我们需要调用这两个方法，但是如果在二者之间出现了异常，很有可能其中一个方法没有执行，这样造成的后果非常严重，可能某个账户减少或者某个账户增加，而另外一个无变化。处理这种方法我们最为理想的思路就是两个方法能够进行绑定，保证二者必须同时成功或者失败，这样才能保证程序能够正常运行。

  事务管理能够保证程序的完整性和一致性，事务有四个特性ACID

  • 原子性（Atomicity）：事务是一个原子操作，由一系列动作组成。事务的原子性确保动作要么全部完成，要么完全不起作用。
  • 一致性（Consistency）：一旦事务完成（不管成功还是失败），系统必须确保它所建模的业务处于一致的状态，而不会是部分完成部分失败。在现实中的数据不应该被破坏。
  • 隔离性（Isolation）：可能有许多事务会同时处理相同的数据，因此每个事务都应该与其他事务隔离开来，防止数据损坏。
  • 持久性（Durability）：一旦事务完成，无论发生什么系统错误，它的结果都不应该受到影响，这样就能从任何系统崩溃中恢复过来。通常情况下，事务的结果被写到持久化存储器中。

事务概念

  事务隔离

  事务隔离实际上属于数据库技术，简单来说即对于事务的处理严谨程度。事务隔离的程度能够避免很多事物问题，但是过高的事务隔离会让程序变的缓慢，类似锁技术，若一个程序锁过于多，会造成程序太慢。事务问题一般分为以下几种。

  事务问题

  • 脏读：事务A读取了事务B更新的数据，然后B回滚操作，那么A读取到的数据是脏数据(事务A读取了事务B未提交的数据)
  • 不可重复读：事务 A 多次读取同一数据，事务 B 在事务A多次读取的过程中，对数据作了更新并提交，导致事务A多次读取同一数据时，结果 不一致。(事务A读取到事务B已经提交的数据)
  • 幻读：事务A在读取事务B时，事务B进行了插入数据，事务A读取完后发现有一条数据未读取，这就叫幻读。(事务A读取到了事务B中未存在的数据)

  事务隔离界别

  • 读未提交（read-uncommitted）会发生所有的事务问题，数据处理速度最快
  • 不可重复读（read-committed）能够避免脏读的情况，数据处理速度次之
  • 可重复读（repeatable-read）能够避免脏读和不可重复读，数据处理更慢
  • 串行化（serializable）能够避免所有的事务问题，但是效率最低

  Spring中事务传播行为

  什么是事务传播行为？假如现在有一个方法Dao，该方法已经绑定事务，此时Service方法中调用了该方法，那么在Spring框架中，该事务同样能够传播。如果Service中还有别的方法，能够通过传播行为的设置，将其嵌套为同一个事务。

  上图就是在Spring事务中的七种传播方式。该图片来自Spring的官方文档，通俗来讲如下

  • PROPAGATION_REQUIRED ：required , 必须。默认值，A如果有事务，B将使用该事务；如果A没有事务，B将创建一个新的事务。
  • PROPAGATION_SUPPORTS：supports ，支持。A如果有事务，B将使用该事务；如果A没有事务，B将以非事务执行。
  • PROPAGATION_MANDATORY：mandatory ，强制。A如果有事务，B将使用该事务；如果A没有事务，B将抛异常。
  • PROPAGATION_REQUIRES_NEW ：requires_new，必须新的。如果A有事务，将A的事务挂起，B创建一个新的事务；如果A没有事务，B创建一个新的事务。
  • PROPAGATION_NOT_SUPPORTED ：not_supported ,不支持。如果A有事务，将A的事务挂起，B将以非事务执行；如果A没有事务，B将以非事务执行。
  • PROPAGATION_NEVER ：never，从不。如果A有事务，B将抛异常；如果A没有事务，B将以非事务执行。
  • PROPAGATION_NESTED ：nested ，嵌套。A和B底层采用保存点机制，形成嵌套事务。

  同时Spring对于数据库的隔离级别提供了自己的设置，能够保证我们在使用事务时正确选择合适的隔离级别

  有一点需要注意的是，MySQL默认的隔离级别为可重复读，Oracle的默认隔离级别为读已提交。

代理模式

  代理模式是一种在框架中被使用十分广泛的模式。代理模式(Proxy Patten)是通过接口或者继承等方法，将我们使用的业务类托管给代理，通过代理来进行部分功能增强(权限管理、日志管理等)。 在理解代理模式之前，首先要掌握Java中的继承多态、反射技术与接口实现。

  什么是代理模式?

  日常生活中，我们需要买一套房屋，如果只由我们自己来办理，需要与经销商沟通，办理各种手续等等。而如果我们将这件事委托给中介，我们只需要告知中介我们的需求，中介就会替我们完成，同时中介还能帮我们提供更多的服务。这种方式我们就能够看作代理模式。

  在使用代理模式时，我们将某个类托管给代理，代理会按照我们的要求为我们添加附加的功能，同时还能提供多个功能组合的实现。代理模式分为静态代理和动态代理。

静态代理

  静态代理一般有两种实现方式继承与聚合

  • 继承 ：实体类实现某个接口，代理类继承该实体类
  • 聚合 ：实体类与代理类实现同一个接口，代理类中维护该接口引用

动态代理

  相比较静态代理需要在运行前就将代理类编写完成，动态代理优势在于能够在运行时自动生成代理类，也就是说，无论我们想要将哪个类实现代理，只需要将该类传参进动态代理的某个方法中，该方法将通过反射以及编译，为程序在运行时自动生成一个代理类

  动态代理在运行时生成代理类的步骤抽象为以下几步

  • 通过I/O技术，解析客户类的源码
  • 将改源码按照同样的格式以字符串的形式保存
  • 将需要实现的增强同样以字符串的形式进行拼接
  • 在通过I/O技术讲该字符串以.java的形式写入磁盘
  • 使用JDK对该文件进行加载，获得class文件
  • 加载完成，获得代理类对象，操作代理类对象即可完成
