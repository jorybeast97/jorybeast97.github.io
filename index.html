Z Garbage Collector染色指针及读屏障总结

发表于 2020-06-07 更新于 2020-06-08 分类于 JVM学习笔记

JVM垃圾收集器的发展过程从某种角度来说可以看作人类在不断追求STW尽量短暂的过程，而这个过程中涌现了很多优秀的垃圾收集器。从开始的单线程 Serial GC到 CMS 再到 G1，直到 ZGC的出现让GC提升到了一个新的高度。虽然官方已经在JDK11中加入了ZGC并且成为了标配，但是其神秘的面纱一直未被揭开，以至于很多人还对他有些陌生。

如果说CMS的出现大大缩短了STW的时间，实现了第一款伪并发GC收集器，那么G1就是一次更大的跨越，G1中提出的Region概念让垃圾回收变得可期待，人们能够按照自己的意愿去尽量缩短STW的时间，虽然这种短时间是以吞吐量为代价，但是在计算机硬件性能十分过剩的今天，这种代价明显是可接受的。

虽然 G1很美好，并且也是目前使用非常多的GC收集器，但是还是有不少的缺点。

  • 由于G1使用三色算法，不可避免地会产生浮动垃圾
  • G1同样无法避免Full GC，那么当堆十分巨大地时候(上百GB)，STW的停顿时间同样是不可忽略的
  • 由于使用了Remember Set记录跨代引用信息，会产生1% ~ 20%的额外内存消耗

如果说 G1 是现在流行的收集器，那么 GC 就可以说是来自未来的怪兽。为什么这么形容 ? 先来看一张 SPECjbb给出的测评结果。

一目了然，ZGC的强大之处在于，无论堆有多大，其GC停顿时间都没有超过10ms。

一图胜千言，同样，该机构也做了很多的测评结果，都发布在The Z Garbage Collector - An Introduction这一文档中。

文档地址 : http://cr.openjdk.java.net/~pliden/slides/ZGC-FOSDEM-2018.pdf

如果足够细心，结合谷歌上的一些资料以及该文档，那么ZGC的学习曲线会变得更加平滑，这一点是来自我的体会，因为ZGC的学习思路和难度相较于之前的GC收集器，都有一个比较大的增长，而更让我们不易理解的是，ZGC很多内容和操作系统相关，如果基础不够扎实，很容易导致概念架空而不得解的情况。

Netty Reactor模型

发表于 2020-06-06 分类于 Web框架

Reactor模型概览

The reactor design pattern is an event handling pattern for handling service requests delivered concurrently to a service handler by one or more inputs. The service handler then demultiplexes the incoming requests and dispatches them synchronously to the associated request handlers.[1]

反应器模式（Reactor pattern）是一种为处理服务请求并发 提交到一个或者多个服务处理程序的事件设计模式。当请求抵达后，服务处理程序使用解多路分配策略，然后同步地派发这些请求至相关的请求处理程序。

先来回忆一个场景，当出现一个网络连接，并发送数据时，都应该经历哪些步骤 ？

一般来说，需要接收连接，读取数据，解码，执行业务逻辑，编码，发送这几个阶段。而如果将所有阶段都集中处理，会让整个逻辑链变得复杂而不可拓展。因此可以将各个步骤交由不同模块来完成，有些类似于责任链。

而 Reactor Pattern 实际上就是对上述场景提出的一种设计模式，既然是设计模式，就是十分高度抽象的，一般来说，最简单的Reactor中应该具有以下几个部分。

  • Reactor : 负责处理IO事件,将事件分发给绑定的Handler处理
  • Handler : 事件处理器,负责对某些事件进行具体的业务处理
  • Acceptor : 负责接收并将事件注册进入Reactor，并交由Reactor处理

该图主要是最简单的Reactor模型，而实际开发和Netty框架中，Reactor设计更为精妙和繁杂，增添了诸多细节例如NIO事件驱动，多Reactor线程池以及各类编解码器。

CAP原则与Eurake和Zookeeper注册中心

发表于 2020-05-23 分类于 分布式与微服务

CAP原则

CAP原则又称CAP定理，指的是在一个分布式系统中，一致性（Consistency）、可用性（Availability）、分区容错性（Partition tolerance）。

  • 一致性 : 所有的分布式节点都要保证数据一致性
  • 可用性 : 在集群中一部分节点失效后,剩余节点还能提供同样的服务
  • 分区容忍性 : 当网络出现问题时,服务出现分区的现象

这三个概念所介绍的就是常见的分布式系统中会经常遇到的问题。
